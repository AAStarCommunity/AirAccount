## ⚡ Core Architecture Principle

**🔒 ALL KEY OPERATIONS MUST BE DONE IN TA (Trusted Application)**

This KMS implementation is **strictly TA-based**, using only the eth_wallet Trusted Application for all cryptographic operations. Zero independent crypto allowed in host.

## 🛡️ eth_wallet TA Foundation

### TA Commands (4 total):
Based on actual eth_wallet TA source code analysis:

```rust
pub enum Command {
    CreateWallet = 0,    // Generate HD wallet with BIP39 mnemonic
    RemoveWallet = 1,    // Securely delete wallet from TEE storage
    DeriveAddress = 2,   // BIP32 HD address derivation
    SignTransaction = 3, // EIP-155 transaction signing
}
```

### TA Input/Output Types:
```rust
// CreateWallet
Input: CreateWalletInput {}
Output: CreateWalletOutput { wallet_id: Uuid, mnemonic: String }

// RemoveWallet
Input: RemoveWalletInput { wallet_id: Uuid }
Output: RemoveWalletOutput {}

// DeriveAddress
Input: DeriveAddressInput { wallet_id: Uuid, hd_path: String }
Output: DeriveAddressOutput { address: [u8; 20], public_key: Vec<u8> }

// SignTransaction
Input: SignTransactionInput { wallet_id: Uuid, hd_path: String, transaction: EthTransaction }
Output: SignTransactionOutput { signature: Vec<u8> }
```

## 🏗️ Technical Architecture

### Host Application (kms-api):
```
┌─────────────────────────────────────┐
│           Host Application          │
├─────────────────────────────────────┤
│ • HTTP Server (AWS KMS compatible)  │
│ • TA Client (optee-teec)           │
│ • Metadata Storage (non-sensitive)  │
│ • JSON ↔ Bincode conversion         │
│ • NO independent crypto operations  │
└─────────────────────────────────────┘
                    │
                    ▼ OP-TEE Client API
┌─────────────────────────────────────┐
│          eth_wallet TA              │
├─────────────────────────────────────┤
│ • BIP39 mnemonic generation        │
│ • BIP32 HD key derivation          │
│ • EIP-155 transaction signing       │
│ • Secure storage (private keys)     │
│ • All crypto operations             │
└─────────────────────────────────────┘
```

### Data Flow:
```
AWS KMS Request → JSON Parser → TA Client → OP-TEE → eth_wallet TA
                                                              │
AWS KMS Response ← JSON Formatter ← TA Client ← OP-TEE ← [crypto result]
```

## 🔒 Security Model

### 🛡️ TEE-Protected (Never leaves TA):
- **Private Keys** - Generated and stored in TA secure storage
- **Mnemonic Seeds** - Generated in TA, returned once on creation
- **HD Derivation** - BIP32 path-based derivation in TA
- **Transaction Signing** - EIP-155 signing with replay protection
- **Cryptographic Operations** - All secp256k1, keccak256, etc.

### 🌐 Host-Stored (Non-sensitive only):
- **Account Metadata** - UUIDs, descriptions, creation timestamps
- **ARN Mappings** - AWS KMS compatibility identifiers
- **API State** - HTTP server state, request parsing

### ❌ Forbidden in Host:
- Private key generation
- Private key storage
- Signing operations
- Mnemonic handling
- Independent crypto libraries

### Error Handling:
- **TAException** - TA communication or operation errors
- **ValidationException** - Invalid request parameters
- **NotFoundException** - Account not found in metadata
- **InternalFailureException** - Serialization/system errors
