# 后端职责拆解 + KMS/CA 承载能力评估

> 类型：架构评估（Architecture Evaluation）
> 触发：YAA（Yet Another Account）—— 一个前端应用，却自起了一个后端，承担了 5 项职责
> 日期：2026-06-19
> 关联：客户端化 / 静态导出方向、DVT 独立性命门、`docs/design/attestation-trust-root-decision.md`

---

## 0. 一句话结论

YAA 自起的后端**基本是在重造 AAStar 已有的栈**。它的 5 项职责应该拆开各归其位，而**不是**一股脑塞进 KMS：
只有 ①（email↔账户**绑定**）真正属于 KMS/CA；②留薄、③搬客户端、④归 SuperRelay、⑤搬客户端。
拆完之后 YAA 自己的后端应趋近于零。

承载能力上：**2GB/8GB 的 i.MX93 足以支撑 ~100 用户、5–10% 并发**的「身份绑定 + 会话 + 轻历史 + 偶发签名」，
前提是**保持薄**（不上重索引、不在板上跑邮件投递）+ **日志轮转**（当前磁盘 83% 已是第一个会先撞墙的约束，不是内存）。

---

## 1. 背景：YAA 后端的 5 项职责

| 职责 | YAA 现状 |
|---|---|
| ① email→账户映射 + 会话 | 自己存映射、发会话 |
| ② 链下索引 / 历史 | 自建索引/历史服务 |
| ③ 签名编排（BLS） | 后端做 BLS 编排 |
| ④ 密钥代理 | 持 bundler/RPC key 的薄 proxy |
| ⑤ 门户只读 | 后端代读链 |

架构直觉（正确）：**一个应用不该自起后端，而应依赖基础服务**。问题是——这 5 项里哪些该并入 KMS（CA 侧），哪些另有归属。

---

## 2. 架构评估：5 项职责的正确归属

| 职责 | 正确归属 | 理由 | 是否进 KMS/CA |
|---|---|---|---|
| ① email↔账户绑定 + 会话 | **KMS/CA（绑定）** + 外部托管邮件服务（OTP 投递） | KMS 定位就是「身份与密钥底层」，绑定天然属于它（passkey 已经在 KMS）。但**发邮件不该跑在 TEE 板上**——SMTP/投递/反滥用是另一类运维负担，用托管邮件服务（SES/Resend/…），板子只存**验证后**的 email↔账户绑定（存哈希）。 | ✅ 绑定进，投递不进 |
| ② 链下索引 / 历史 | KMS 现有的**薄 `tx_log`** + subgraph/indexer 或客户端缓存（重的部分） | KMS 已有 `tx_log` 表和 `/stats`，这就是「留个薄服务方便」。**重索引绝不上板**——2GB 放不下 graph-node + Postgres。 | ⚠️ 仅保留已有薄层，不扩 |
| ③ 签名编排（BLS） | **客户端** | DVT 独立性命门：后端做 BLS 编排 = 「CA 式中间人」，正是静态化方向要去掉的。**KMS 本来就零 BLS 代码**（已确认）。永不上 KMS。 | ❌ 搬客户端 |
| ④ 密钥代理（bundler/RPC key） | **SuperRelay**（AAStar 已有的 bundler 网关）/ 公共端点 | KMS 管 TEE 密钥托管，不该掺 RPC 代理。AAStar 栈里 **SuperRelay 就是干这个的**（`eth_sendUserOperation` 等 + 双签）。 | ❌ 归 SuperRelay |
| ⑤ 门户只读 | **客户端**直接读链 + subgraph | 根本不是后端职责（registry 原 app 就是前端直读链）。 | ❌ 搬客户端 |

### 拆完之后的目标架构

```
YAA 前端
  ├── 身份 / 密钥 ──────────→ AirAccount KMS（① 绑定 + passkey + TEE 签名）
  ├── bundler / RPC ───────→ SuperRelay
  ├── gas 赞助 ─────────────→ SuperPaymaster
  ├── 读 / 历史 ────────────→ subgraph / 公共 indexer（+ 客户端缓存）
  └── 签名编排（BLS）───────→ 客户端本地（DVT 独立性）

YAA 自己的后端 → 趋近于零（顶多一个不持密钥、不存数据的薄 BFF）
```

**回答「剥离到 KMS 是否合理」**：合理的只有 ①（绑定部分），且要把邮件**投递**外置。其余四项各有更合适的归属，把它们都塞进 KMS 是错的——会让本该越来越薄的安全内核重新变胖，反而背离客户端化/DVT 方向。

## 3. 承载能力评估（i.MX93，实测）

### 3.1 实测硬件 / 占用（生产板 192.168.2.50，2026-06-19）

| 维度 | 实测值 | 解读 |
|---|---|---|
| CPU | 2× Cortex-A55 | TEE 密码学吞吐的限制项 |
| 内存 | 1.9 GB 总；**空闲仅用 225 MB / 可用 1.7 GB** | 极宽裕 |
| `kms-api-server` 常驻 | **RSS ≈ 7.4 MB** | 服务本身几乎不吃内存 |
| 磁盘 | 8.2 GB 总，**6.4G 已用 / 1.4G 空闲 / 83%** | ⚠️ **第一个会撞墙的约束**（系统镜像 + 二进制备份 + 日志，不是业务数据） |
| DB | 121 钱包 = **299 KB**（≈ 2.5 KB/钱包） | 业务数据极小 |

### 3.2 实测延迟（LAN 直连 :3000，隔离板子算力）

| 操作 | 延迟 | 说明 |
|---|---|---|
| `/health`（板上 localhost） | **~2 ms** | 非 TEE 读，极快 |
| `/QueueStatus` | ~48 ms（含 LAN） | 非 TEE |
| `CreateKey`（同步返回） | **220 ms** | 派生（PBKDF2+BIP32）是**后台异步**的，创建本身快 |
| 派生到 `ready` | ~秒级（seed 缓存后更快） | 一次性，每个 key 只付一次 |

> 签名（Sign）需要 WebAuthn assertion，未单独压测；架构上 = CA 侧 assertion 验签（快）+ TEE 内 ECDSA（seed 缓存后数十 ms）。

### 3.3 容量推算：100 用户 @ 5–10% 并发

并发 = 5–10 个同时活跃会话。

- **内存**：服务 7.4 MB + 每连接 tokio 开销（KB 级）→ 10 并发可忽略。内存能撑的远不止 100 人（数千人的身份数据也是 MB 级）。**内存非约束。**
- **存储（业务数据）**：100 用户身份+钱包 ≈ 250 KB，1000 用户 ≈ 2.5 MB。**业务数据非约束**；撞墙的是系统镜像 + 日志增长 → 必须 **logrotate**。
- **CPU / TEE 吞吐（真正的限制项）**：
  - 非 TEE 读 ~2 ms → 单核每秒数百~上千次，读毫无压力。
  - TEE 写/签（CreateKey/Sign）经 TA 串行化。CreateKey 同步 220 ms；Sign 若数十~百余 ms，单线程 TA → 粗估 **~7–20 次/秒上限**。
  - 100 用户 @ 5–10% 并发、**偶发**签名，平均需求 < 1–2 签/秒，**远低于上限**；突发由队列吸收（已有 `QueueStatus` + 限流）。

### 3.4 结论

> **2GB/8GB i.MX93 足以承载 ~100 用户、5–10% 并发**的身份绑定 + 会话 + 轻历史 + 偶发签名。
> 先撞墙的不是内存（极宽裕），而是 **(a) 磁盘余量**（83%，需 logrotate + 清理旧二进制备份）和 **(b) 突发签名下的 TEE 吞吐**。

## 4. 瓶颈与扩展路径

| 若需要 | 怎么扩 | 不要怎么扩 |
|---|---|---|
| 更多用户 / 更多身份数据 | 现板即可（数据 MB 级） | —— |
| 更多磁盘余量 | logrotate + 清旧二进制备份；16GB/**256GB** 升配 | —— |
| 更高签名吞吐 | **横向扩**：多块板做多个独立 KMS 节点 | ❌ 升配单板（16GB 内存不提升 TEE 吞吐，ELE/CPU 不变） |

**关键洞察**：单板内存/存储升配**不解决签名吞吐**（瓶颈是 ELE/CPU，不是 RAM）。要更高吞吐就**横向加节点**——这恰好与 DVT / 去中心化方向一致：不是把一块板做大，而是让多个独立节点并存（也分散了「单点信任」）。

---

## 5. 行动建议

**架构（YAA 侧，提 issue 到 YAA 仓库，本仓不代改）**：
1. 砍掉 ③（BLS 编排）→ 客户端；④ → SuperRelay；⑤ → 客户端直读链。
2. ② 重索引 → subgraph；只在需要时调 KMS 的薄 `tx_log`。
3. ① email 绑定 → 依赖 KMS（见下），邮件投递走托管服务。

**KMS 侧（本仓，承接 ① 时再做，非本月）**：
1. email↔账户绑定端点（存**哈希**，不存明文邮箱）；邮件 OTP 投递**外置**为可插拔 provider，不在板上跑投递守护进程。（与 issue #58 email OTP 合并考虑。）
2. 运维硬约束：**logrotate**（磁盘 83%）；清理 `kms-api-server.bak-*` 旧二进制。
3. 容量超出单板时：横向加 KMS 节点，而非升配单板。
