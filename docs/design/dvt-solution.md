## 0. 一句话与诚实边界

**一句话**：DVT 是 AirAccount 安全体系里**独立于 owner key 的第二道闸门 + 独立验证层**——一组链下独立验证节点，对大额/高风险操作按各自策略门限共签，使"单把 key 被盗 = 全损"退化为"必须同时攻破 owner key + 门限内的独立策略节点才损"。

**诚实边界（先讲清楚，贯穿全文）**：

> **DVT 的安全增益完全来自「独立性」，不来自「再签一次」。** 若 DVT 节点盲签（拿到请求就签、不跑独立策略、私钥与 owner key 同处一环境），则攻击者控制 owner key + CA 后只需再骗 DVT 盲签一次 → DVT 形同虚设，沦为橡皮图章。真正价值的全部来源是 DVT 拥有 owner key 之外、CA 改不了的**独立信号**（独立私钥 + 独立策略 + 独立通道）。本文所有"增益""量级"论述都以这个前提为锚。

## 1. DVT 是什么

### 1.1 定义（业务语言）

DVT = Distributed Validator Technology = 一组**独立的链下验证节点**（参考实现见 YetAnotherAA-Validator）。每个节点：

- 持有**自己的 BLS 私钥**（彼此独立，不共享，理想情况下也不与 owner key 同处一个 TEE / 设备）。
- 收到操作请求时，**各自独立按策略判断**该不该批准。
- 同意则产出一个**部分签名（partial signature）**。
- 当 ≥ 门限个（≥ N of M）独立节点都同意，这些部分签名被 **BLS12-381 聚合**成一个聚合签名。
- 链上 / 验证侧只需校验："这个聚合签名对应 ≥ 门限个**已注册** DVT 节点" → 通过则视为 DVT 网络共同批准了这笔操作。

### 1.2 业务定位

在 AirAccount 安全体系里，DVT 占据两个互不相同的位置（详见 §3）：

```
┌──────────────────────────────────────────────────────────┐
│  AirAccount 安全体系                                        │
│                                                            │
│   owner key 层  ── passkey (WebAuthn) + TEE 私钥           │
│        │            "你是不是密钥的合法持有者"              │
│        │                                                   │
│   DVT 层  ────────  ① 独立 co-signer：BLS 门限共签          │
│        │             "这笔操作是否经独立节点按策略批准"     │
│        │            ② 验证者：验 passkey / attestation     │
│        │             "请求 / 签名是否来自真 TEE"            │
│        ▼                                                   │
│   纵深防御：单把 key 被盗 ≠ 全损                            │
└──────────────────────────────────────────────────────────┘
```

一句话定位：**DVT 是 AirAccount 安全体系的「独立第二闸门（授权维度）+ 独立验证层（验证维度）」**。

## 2. DVT 给安全体系的增量

### 2.1 解决的单点风险

| 单点风险（无 DVT 时） | DVT 如何缓解 |
|---|---|
| **owner key 被盗 / 钓鱼**（passkey 被诱导、私钥泄露） | 即使拿到 owner key，攻击者还得让 ≥ 门限个独立 DVT 节点也批准；DVT 有独立策略（限额 / 白名单 / 额外确认），不盲目批 |
| **TEE 被攻破 / 假 TEE**（威胁模型向量 5：KMS 不是真 NXP 芯片） | co-signer **不依赖 attestation 信任根**——它不验"签名来自真 NXP 芯片"，只要"≥ 门限独立节点按策略批准"。假 TEE 骗不了独立节点的独立策略 |
| **CA（KMS Host）作恶 / 被接管** | 独立策略在链上 / 节点本地，CA 改不了；独立确认通道直连用户另一设备、不经 CA |

### 2.2 价值评估（主架构师定调）

把安全模型从：

```
owner key 单点被盗  =  全损
```

升级为：

```
owner key 被盗  +  攻破门限内的独立策略  =  才损
```

**增益量级 = 独立性 × 策略强度 × 门限**

- **独立性**：DVT 私钥 / 策略 / 通道相对 owner key 与 CA 的独立程度。**这是上限决定项**——独立性为 0（盲签、同环境）则增益为 0。
- **策略强度**：限额、白名单、额外确认等策略对攻击的实际约束力。
- **门限**：N of M 中 N 越高、M 越多样，串谋越难。

**上限取决于 DVT 独立于 CA 的程度**。因为引入了延迟、复杂度、激励成本，**不对所有操作上 DVT，只对大额 / 高风险分层启用**（如 > $1000）。这是一个工程权衡：小额操作走 owner key 即可，大额操作才付得起 DVT 的延迟与成本。

## 3. DVT 的两个角色（主架构师定的框架）

DVT 不是单一功能，而是两个正交角色。**混淆这两个角色会导致错误的安全推理**（尤其在它们与 R-1 的关系上）。

### 角色 A — 验证者（Verifier）

- **做什么**：验证 passkey ceremony / TEE attestation，确认"请求 / 签名响应确实来自真实硬件、未被篡改"。
- **防什么**：CA 作恶、假 TEE 在"**验证**"维度的伪造。
- **与 R-1 的关系**：**卡 R-1**。验证者要断言"签名来自真 NXP 芯片"，就绕不过 attestation 信任根（R-1：ELE 设备 key 是否连 NXP 根）。R-1 未收口前，验证者角色的"真硬件"判断只能退化为 TOFU / 自签登记（见 #37 §9）。

### 角色 B — 独立 co-signer

- **做什么**：BLS 门限共签 + 各自跑独立策略，对操作做独立**授权**。
- **防什么**：owner key 被盗 / 钓鱼 / 单点失陷。
- **与 R-1 的关系**：**不卡 R-1**（关键洞察）。co-signer **不验**"签名来自真芯片"，只断言"≥ 门限个独立节点按策略批准了这笔操作"。它的安全基础是**独立性**，不是 attestation 信任根。

> **关键洞察（为什么 co-signer 不卡 R-1 如此重要）**：
> 即使 R-1 永远没解、即使 KMS 是假 TEE（威胁模型向量 5），**大额操作仍需独立 DVT co-signer 共签 → 假 TEE 骗不了独立节点的独立策略**。所以 **co-signer DVT 是威胁模型向量 5 在 R-1 收口前的现实缓解手段**。它把"必须先解决 attestation 信任根才能防假 TEE"这个死结，绕成了"用独立授权层先兜住大额风险"。

| | 角色 A 验证者 | 角色 B co-signer |
|---|---|---|
| 维度 | 验证（这是不是真硬件） | 授权（该不该批准这笔操作） |
| 防 | CA 作恶 / 假 TEE 的伪造（验证侧） | owner key 被盗 / 单点失陷 |
| 信任基础 | attestation 信任根（NXP） | 节点独立性（私钥 + 策略 + 通道） |
| 卡 R-1？ | **是** | **否** |
| 现状落地难度 | 高（依赖 #37 / R-1） | 中（依赖独立性工程，不依赖 NXP） |

## 4. 依赖与安全上下游 / 假设

### 4.1 命门假设：独立性

> **独立性假设是 DVT 的命门。** 它一旦不成立，DVT 的全部安全价值清零。三条独立性，缺一即降级：

1. **独立私钥**：DVT 私钥不在 owner key 的同一个 TEE / 同一台设备 / 同一运营者手里。否则攻破一处 = 同时拿到两层。
2. **独立策略（CA 改不了）**：限额 / 白名单 / 额外确认等策略落在**链上合约**或**节点本地**，CA（KMS Host）无权修改。否则 CA 被接管即可把策略改成"全批"。
3. **独立通道（不经 CA）**：额外确认直连用户的**另一台设备**，不经过 CA。否则中间人 / CA 可代答确认。

### 4.2 防 DVT 自身被攻破

- **门限（≥ N of M）**：单个节点失陷不致命，需同时攻破 ≥ N 个。
- **多样性**：不同运营者、不同辖区 / 法域、不同软件栈。同质化节点会被同一漏洞 / 同一法律强制一锅端，门限形同虚设。

### 4.3 激励必须绑「正确执行策略」

> **激励绑错 = 毁掉安全价值。** 若激励绑"签名次数"，节点为了攒积分会盲签 → 独立策略名存实亡。

- 激励必须绑**正确执行策略**，而非签名次数。
- 上游：对接 **SuperPaymaster v5 `ROLE_DVT`**（角色注册 / 质押 / 退出 / 惩罚）+ **PGL 贡献记录**（链上声誉 SBT，不可转让，记"正确执行"而非"签了多少次"）。
- **盲签 / 签错可被 slash**：经济惩罚使"为积分盲签"无利可图。

### 4.4 与各层的关系（上下游）

| 关系对象 | 关系 |
|---|---|
| **owner key 层（passkey + TEE 私钥）** | **互补、纵深**。owner key 验"你是不是合法持有者"，DVT 验"该不该批 / 是不是真硬件"。两层独立失陷才全损。 |
| **KMS（TEE 私钥管理）** | co-signer 补 KMS 在威胁模型向量 5（假 TEE）下的缺口；验证者与 KMS attestation **互证**（见 §8）。 |
| **R-1（attestation 信任根，#37）** | 验证者角色卡 R-1；co-signer 角色不卡 R-1。详见 §3。 |
| **SuperPaymaster v5 `ROLE_DVT` / PGL** | 上游激励与惩罚基础设施，使独立策略有经济约束力。 |

## 5. 工作原理 + 技术机制

参考实现：**YetAnotherAA-Validator**（NestJS signer service + Foundry 验证合约，BLS12-381 / @noble/curves，链上 EIP-2537 precompile 验证）。

### 5.1 端到端流程（co-signer 主路径）

```
用户发起大额操作 (> 阈值)
   │
   ▼
owner key 层先签 (passkey / TEE)         ← 第一道闸门
   │
   ▼
协调者把"请求包"广播给一组 DVT 节点
   │  请求包 = 原始操作 + owner 签名 + 业务摘要 + nonce
   ▼
每个 DVT 节点【独立】执行：
   ① 验请求完整性 (owner 签名是否有效)
   ② 跑【独立策略】(链上合约 / 本地)：
        - 限额 (这笔金额是否超限)
        - 白名单 (收款地址是否允许)
        - 额外确认 (经【独立通道】问用户另一设备)
   ③ 通过 → 用自己的 BLS 私钥对请求哈希签 → partial signature
   │
   ▼
收集到 ≥ 门限个 partial signature
   │
   ▼
任意节点做【BLS 聚合】→ 1 个聚合签名 + 聚合公钥 (EIP-2537 格式)
   │
   ▼
链上 / 验证侧校验：
   verifyAggregateSignature(...) → "≥ 门限个【已注册】DVT 节点签了"
   │
   ▼
通过 → 第二道闸门放行；任一环节失败 → 拒绝
```

### 5.2 关键机制

| 机制 | 说明 | 参考实现对应 |
|---|---|---|
| **BLS12-381 聚合共签** | 每个节点产 partial signature，可数学聚合成单签名；验证成本与签名数弱相关（~450k gas / 3 signers） | `modules/bls/`，`POST /signature/sign`、`POST /signature/aggregate` |
| **合约验门限** | 链上合约维护已注册节点公钥表，校验聚合签名对应 ≥ 门限个注册节点 | `AAStarValidator.sol`：`registerPublicKey` / `isRegistered` / `verifyAggregateSignature`，链上 EIP-2537 |
| **独立策略引擎** | 每节点独立查策略（理想：链上策略合约，CA 改不了），不盲签 | 参考实现尚弱（见 §7），需补 |
| **独立确认通道** | 额外确认直连用户另一设备，不经 CA | 参考实现未含，需补（见 §7） |
| **注册 / 质押 / slash** | 节点链上注册公钥；质押 + 惩罚保证诚实 | 参考实现有链上注册（`/node/register`）；质押 / slash 对接 SuperPaymaster `ROLE_DVT` |
| **节点发现 / 通信** | P2P gossip（WebSocket）做节点发现与签名收集 | `modules/gossip/`，`/ws` |
| **节点身份独立** | 每实例独立 BLS key pair + 本地状态文件，互不知道彼此私钥 | `node_*.json`（含私钥，绝不出 API；生产用 KMS 托管） |

## 6. 安全原理

### 6.1 为什么提升安全：独立性 → 攻击面从单点变门限

无 DVT：攻击面 = **1 把 owner key**。攻破它 = 全损。

有（真正独立的）DVT：攻击面 = **owner key + 门限内独立策略节点**。攻击者必须：

1. 拿到 owner key（钓鱼 / TEE 失陷），**且**
2. 让 ≥ N 个独立节点也批准——而它们：
   - 私钥不在同一环境（攻破一处拿不到第二层）；
   - 策略 CA 改不了（接管 CA 也改不了限额 / 白名单）；
   - 额外确认走独立通道（中间人代答不了用户另一设备）。

攻击面从"单点"扩成"门限 × 独立维度"，攻击成本指数上升。

### 6.2 为什么盲签 = 废

若节点盲签（拿请求就签、无独立策略、私钥同环境）：攻击者控制 owner key + CA 后，只需把请求转发给盲签节点，节点照签 → DVT 加不了任何约束。**此时 DVT 只是"会盲签的橡皮图章"，徒增延迟与成本，零安全增益。** 这就是 §0 诚实边界的技术根据。

### 6.3 为什么 co-signer 不卡 R-1

R-1 是"如何密码学证明签名来自真 NXP 芯片"的信任根缺口（#37）。**co-signer 根本不问这个问题**——它问的是"≥ 门限个独立节点是否按策略批准"。

- 验证者角色才需要"真硬件"断言 → 卡 R-1。
- co-signer 角色只需要"独立授权" → 与 attestation 信任根**正交**。

因此 **co-signer 在 R-1 未收口、甚至 KMS 是假 TEE 时仍然有效**，这是它作为威胁模型向量 5 现实缓解手段的根本原因。

## 7. 理想设计 vs 当前雏形

### 7.1 当前雏形（用户描述的基础共签）

- 一组独立 BLS 节点（YetAnotherAA-Validator 那些），各持私钥。
- 大额操作时节点对操作 BLS 聚合共签。
- 合约验"≥ 门限个注册 DVT 节点签了"。
- "DVT 因子" = 这笔操作拿到 N 个独立节点共签批准，独立于 owner key 的第二道闸门。

**雏形的不足**：偏重"再签一次"，独立策略 / 独立通道 / 激励绑定 / slash 尚弱或缺位——按 §6.2，**离"盲签橡皮图章"只差一步**。

### 7.2 理想设计（要补的）

主架构师定调的三个关键完善：

1. **补独立策略引擎**：策略落**链上合约**或节点本地，CA 改不了（限额 / 白名单 / 速率）。节点查策略再决定签不签，杜绝盲签。
2. **补独立确认通道**：高风险操作经**独立通道**直连用户**另一台设备**做额外确认，不经 CA。
3. **激励绑「正确执行」+ slash**：对接 SuperPaymaster `ROLE_DVT` + PGL 贡献记录，激励绑正确执行策略而非签名次数；盲签 / 签错可 slash。
4. **与 KMS 分层联动**：小额走 owner key；大额加 DVT co-signer；DVT 验证者与 KMS attestation 互证（§8）。

| 维度 | 当前雏形 | 理想设计 |
|---|---|---|
| 共签 | ✅ BLS 聚合 + 合约验门限 | ✅ 同 + 多样性约束（运营者 / 法域） |
| 独立私钥 | ⚠️ 节点独立，但与 owner key / 运营者独立性未强约束 | ✅ 强约束：不同环境 / 不同运营者 |
| 独立策略 | ❌ 弱（参考实现策略引擎不完整） | ✅ 链上策略合约，CA 改不了 |
| 独立通道 | ❌ 无 | ✅ 直连用户另一设备 |
| 激励 | ⚠️ 链上注册有，质押 / slash 未绑正确执行 | ✅ `ROLE_DVT` + PGL，绑正确执行，可 slash |
| 与 KMS 联动 | ❌ 未分层 | ✅ 分层验证 + attestation 互证 |

## 8. 与 KMS 安全机制的配合 / 互补

DVT 与 KMS 不是替代关系，而是**纵深**：

### 8.1 双层纵深（owner key 层 + DVT 层）

```
owner key 层 (passkey + TEE)   →  "你是不是密钥合法持有者"   →  防冒用
        +
DVT 层 (co-signer + 验证者)     →  "该不该批 / 是不是真硬件"  →  防单点失陷 / 假 TEE
        =
纵深防御：两层独立失陷才全损
```

### 8.2 co-signer 补威胁模型向量 5

KMS 自身无法证明自己是真 TEE（向量 5：假 TEE），且这依赖 R-1 收口。**co-signer 在 R-1 收口前先兜住大额风险**——假 TEE 骗不了独立节点的独立策略。这是 KMS 在 attestation 缺口期最现实的补强。

### 8.3 分层验证（按金额 / 风险）

| 操作 | 验证路径 |
|---|---|
| 小额（< 阈值） | 仅 owner key（passkey / TEE），低延迟 |
| 大额（> 阈值，如 > $1000） | owner key **+** DVT co-signer 门限共签 |
| 高风险（白名单外 / 异常） | owner key + DVT + 独立通道额外确认 |

### 8.4 DVT 验证者 + KMS attestation 互证

- KMS 侧出 attestation evidence（#37：TA 度量 + ELE 设备证据）。
- DVT 验证者角色独立校验该 evidence → 形成"KMS 自证 + 独立第三方验证"的**互证**，比纯 KMS 自证更去中心化。
- ⚠️ 这一路径卡 R-1（验证者断言"真硬件"需 attestation 信任根）；R-1 未收口时退化为 TOFU 互证（见 #37 §9）。co-signer 路径不受此约束。

## 9. 实现路线（MVP → 理想）

| 阶段 | 内容 | 与 #37 / #63 / #68 的关系 |
|---|---|---|
| **MVP** | 部署 ≥ 3 个独立 BLS 节点（YetAnotherAA-Validator）；链上 `AAStarValidator` 验门限；大额操作触发 co-signer 共签 | #63 / #68 落地起点（待对齐）；**不依赖 #37 / R-1** |
| **P1 独立策略** | 上链策略合约（限额 / 白名单 / 速率）；节点查策略再签，杜绝盲签 | 独立于 #37 |
| **P2 独立通道** | 高风险操作经独立通道直连用户另一设备额外确认 | 独立于 #37 |
| **P3 激励 + slash** | 对接 SuperPaymaster `ROLE_DVT`（注册 / 质押 / 退出 / 惩罚）+ PGL 贡献记录；盲签 / 签错可 slash | 上游 SuperPaymaster v5 |
| **P4 多样性** | 引入不同运营者 / 法域 / 软件栈节点，强化门限抗串谋 | 生态 / 治理 |
| **P5 验证者 + attestation 互证** | DVT 验证者角色独立校验 KMS attestation evidence，与 KMS 自证互证 | **依赖 #37 / R-1**（卡信任根；R-1 前退 TOFU） |

> **路线设计要点**：co-signer 路径（MVP / P1 / P2 / P3）**全程不依赖 #37 / R-1**，可独立推进，先把"防 owner key 单点被盗 + 缓解假 TEE"落地；验证者路径（P5）才并入 #37 的 attestation 信任根工作。两条路径解耦，使 DVT 不被 R-1 阻塞。

## 10. 需主架构师拍板的点

1. **DVT 阈值策略**：大额阈值定多少（$1000？分级？按 token / 链区分？）；哪些操作类型强制 DVT。
2. **门限参数**：N of M 的 N、M 取值；多样性约束的硬性程度（是否强制跨运营者 / 跨法域）。
3. **独立策略落点**：策略上链合约 vs 节点本地配置 vs 混合；策略变更的治理流程（谁能改、改了如何防 CA 单方修改）。
4. **独立确认通道形态**：复用 WebAuthn 另一设备？独立 App push？短信 / 邮件（弱）？通道本身的安全等级。
5. **激励 / slash 参数**：质押额度、slash 条件（如何客观判定"盲签 / 签错"）、PGL 贡献记录与 `ROLE_DVT` 的具体映射。
6. **节点运营模型**：自营 vs 社区运营 vs 混合；节点准入 / 退出门槛（关系到独立性 / 多样性这个命门假设能否成立）。
7. **#63 / #68 范围对齐**：本文 MVP 与现有 #63 / #68 issue 的边界、命名、是否合并。
8. **威胁模型文档补建**：`threat-model-ca-adversary.md`（含 §3 两角色论证、向量 5 定义）目前缺失，建议补建并与本文互链。
